# Windows/Linux 桌面端样式调整 Spec

## 1. 背景

当前项目已经具备基础桌面端分栏能力，核心入口位于 `lib/main.dart` 与 `lib/pages/desktop_layout.dart`。现阶段桌面端更多是“移动端页面的横向拼接”，视觉层次、留白、消息区布局、左侧导航形态与参考图仍有明显差异。

本次目标不是新增业务功能，而是将 Windows/Linux 桌面端整体样式调整为接近参考图的产品形态：更轻的桌面应用感、更明确的三段式信息架构、更统一的圆角卡片与浅灰背景体系，以及更贴近桌面端的输入区、消息区和侧边导航表现。

## 2. 目标

将 Windows/Linux 桌面端 UI 调整为参考图所示风格，达到以下结果：

1. 整体界面呈现为桌面应用工作台，而非移动端页面直接放大。
2. 左侧导航、会话列表/智能体列表、右侧详情区形成稳定的桌面三栏结构。
3. 聊天详情区域具备参考图中的居中内容画布、状态卡片、消息气泡、底部输入框样式。
4. 视觉语言统一为浅灰背景、白色卡片、细分隔、较大圆角、弱阴影、轻量图标。
5. 移动端现有布局与交互保持不变。

## 3. 适用范围

### In Scope

- Windows / Linux / MacOS 平台下的主框架布局
- 左侧一级导航栏
- 会话列表页桌面样式
- 智能体列表页桌面样式
- 聊天详情页桌面样式
- 输入区桌面样式
- 空状态、加载状态、选中态、悬停态

### Out of Scope

- 新增业务能力
- 数据结构调整
- 移动端 UI 重构
- Web/iOS/Android 样式同步
- 参考图中的品牌文案、真实任务数据、图片资源一比一复刻

## 4. 当前实现概况

### 4.1 现状

- `lib/main.dart`
  - 通过 `isDesktopOrTabletPlatform(context)` 切换桌面布局
  - 当前桌面端直接挂载 `DesktopLayout`
- `lib/pages/desktop_layout.dart`
  - 已具备左导航 + 列表区 + 详情区的基础结构
  - 使用 `SidebarX` 实现左侧栏
  - 聊天/智能体详情区已具备联动能力
- `lib/pages/chats.dart`
  - 当前列表页仍保留移动端 `Scaffold + AppBar` 结构
- `lib/pages/chat.dart`、`lib/pages/chat/message_list.dart`、`lib/pages/chat/message_input.dart`
  - 聊天内容区仍偏移动端页面组织方式

### 4.2 主要问题

1. 左侧导航视觉风格与参考图差异较大，缺少“桌面工作台”气质。
2. 列表页内部仍是移动端 AppBar 语义，桌面空间利用率不够自然。
3. 详情区缺少统一内容容器、顶部标题区、状态卡片区和底部固定输入区的层级。
4. 颜色、圆角、边框、间距尚未形成一致设计 token。
5. Windows/Linux 桌面端没有与参考图一致的窗口内容区节奏。

## 5. 参考图拆解

参考图可概括为以下视觉特征：

1. 最外层是低饱和浅灰背景。
2. 左侧为窄导航栏，顶部品牌名，底部账户入口，中间是任务/站点/聊天等菜单。
3. 中间主区为宽画布，顶部保留标题区，内容区留白充足。
4. 聊天过程中的“已编辑文件”“运行命令”“已处理时间”等信息以白色圆角卡片或轻量标签呈现。
5. 用户消息与系统消息不是强对比聊天气泡，而是偏文档流式的轻量块级内容。
6. 底部输入区固定吸附在内容底部，输入框较宽、圆角较大、按钮轻量。

本次不要求像素级还原，但要求在布局结构、密度、层次、组件语义上明显接近该风格。

## 6. 设计原则

1. 桌面优先：布局以桌面信息密度和鼠标交互为基础。
2. 轻量克制：减少高饱和色块和过重阴影。
3. 内容优先：让聊天内容成为主视觉中心。
4. 一致性：导航、列表、详情、输入区使用统一圆角、边框和状态色。
5. 渐进改造：尽量复用现有页面状态流转，优先改样式与容器结构。

## 7. 目标布局

## 7.1 总体结构

Windows/Linux 桌面端采用三段式结构：

- 左侧一级导航栏：固定宽度 `88-96`
- 中间列表栏：固定宽度 `320-360`
- 右侧详情内容区：自适应填充

建议整体外边距 `12-16`，主容器圆角 `20-24`。

## 7.2 左侧一级导航栏

目标形态：

- 使用竖向窄侧栏，而非当前较宽的 `SidebarX extended` 风格
- 顶部为应用品牌区，显示应用名或 Logo
- 中段为一级导航图标
  - Chats
  - Bots
  - Profile
- 底部为用户入口/设置入口

样式要求：

- 背景色接近 `#EEF2F5`
- 选中态为白底或浅蓝底圆角胶囊
- 未选中态为透明背景 + 深灰图标
- 图标尺寸 `20-22`
- 导航项高度 `44-48`

实现建议：

- 评估是否保留 `SidebarX`
- 如果 `SidebarX` 难以实现目标密度，改为自定义 `Container + Column + InkWell`

## 7.3 中间列表栏

目标形态：

- 作为独立白色卡片面板存在
- 顶部包含页面标题和主操作按钮
- 搜索框嵌入标题区域下方
- 列表项采用更扁平的桌面列表样式

样式要求：

- 面板背景白色
- 圆角 `20-24`
- 内边距 `16-20`
- 列表项高度控制在 `64-76`
- 选中项需要有明显但不刺眼的背景高亮
- hover 时显示浅灰背景

Chats 页要求：

- 标题区展示“聊天”或本地化标题
- 新建聊天按钮放在标题区右上角
- 搜索框高度 `40-44`
- 空状态插画可保留，但需缩小并与桌面留白匹配

Bots 页要求：

- 与 Chats 页使用同一结构语言
- 新建智能体入口优先用按钮或图标按钮置于标题区

## 7.4 右侧详情区

目标形态：

- 作为主要工作画布
- 背景不应直接铺满纯白，而应为浅灰工作区内嵌白色内容卡片
- 顶部保留上下文标题区域
- 中部为消息流
- 底部为固定输入区

建议结构：

1. 顶层工作区背景：浅灰
2. 中央内容容器：最大宽度 `920-1040`
3. 内容容器内部：
   - 顶部标题/描述区
   - 状态卡片区
   - 消息流
   - 底部输入区

## 8. 聊天详情页样式要求

## 8.1 标题区

顶部应支持：

- 当前会话标题
- 所属智能体名称
- 可选的轻量操作按钮

样式要求：

- 不使用移动端 AppBar 视觉
- 标题区域更像内容页 header
- 上下留白 `20-24`
- 分隔线使用 `1px` 浅灰线或不使用硬分割，优先留白分组

## 8.2 消息流

参考图中消息区更像工作记录流，要求：

- 增加消息容器最大宽度限制，避免超宽文本整屏铺开
- AI 输出使用块状正文排版，段落间距更清晰
- 用户消息可显示为右侧轻量圆角气泡，或保持浅灰小胶囊样式
- 系统过程信息单独使用“状态卡片”样式展示

### 状态卡片定义

适用于：

- 已编辑文件
- 已运行命令
- 正在思考
- 耗时信息
- 工具调用记录

样式要求：

- 白色卡片
- 边框 `1px` 浅灰
- 圆角 `16-18`
- 内边距 `12-16`
- 支持图标 + 主文案 + 次文案

## 8.3 输入区

目标接近参考图底部输入体验：

- 输入区固定在详情面板底部
- 宽输入框，大圆角
- 左侧可放附件/快捷入口
- 右侧发送按钮、语音按钮、模型选择按钮保持轻量

样式要求：

- 输入容器白底
- 外层可有极轻阴影或浅边框
- 圆角 `20-24`
- 高度不低于 `64`
- 与消息流之间保留 `16-24` 间距

## 8.4 空状态

当未选中聊天时，右侧不再只显示简单图标 + 文案，而应升级为桌面空态：

- 顶部保留当前模块标题
- 中央展示轻量插画/图标
- 提示文案更简洁
- 可附带主操作按钮，例如“开始新聊天”

## 9. 视觉 Token 建议

建议在桌面端抽离一组样式 token，避免散落魔法值。

### 9.1 颜色

- App 背景：`#F3F5F7`
- 面板背景：`#FFFFFF`
- 次级面板/输入底：`#F7F8FA`
- 主文字：`#1F2329`
- 次文字：`#6B7280`
- 分隔线：`#E5E7EB`
- hover：`#F3F4F6`
- 选中态：`#E8F0FE` 或同级浅蓝色

### 9.2 圆角

- 一级容器：`24`
- 二级卡片：`20`
- 列表项/状态块：`16`
- 小控件：`12`

### 9.3 阴影

- 仅用于主卡片和输入区
- 强度要弱，避免移动端卡片式厚重阴影

### 9.4 字体层级

- 页面标题：`20-24 / semibold`
- 面板标题：`16-18 / semibold`
- 正文：`14-16 / regular`
- 辅助信息：`12-13 / regular`

## 10. 交互要求

1. Windows/Linux 下悬停态必须明确。
2. 列表选中后，详情区切换应平滑，无整页跳转感。
3. 新建聊天后，右侧直接展示新聊天详情。
4. 新建智能体/编辑智能体维持当前桌面联动逻辑，但样式升级为桌面面板或桌面弹层语言。
5. 输入框获得焦点时应有清晰但克制的高亮反馈。

## 11. 响应式要求

### 11.1 Windows/Linux

- `>= 1280`：完整三栏
- `1024 - 1279`：保留三栏，但列表栏可压缩到 `300`
- `< 1024`：可退化为当前较简桌面布局，或沿用移动/平板逻辑

### 11.2 平台边界

本次 spec 仅强制要求 Windows / Linux。由于当前代码将 `macOS` 也纳入桌面布局，后续实现时需明确：

- 方案 A：样式 token 同步到 macOS
- 方案 B：本轮仅在 Windows/Linux 启用新样式，macOS 暂不调整

默认建议采用方案 A，以减少桌面端分叉。

## 12. 技术实现建议

### 12.1 优先调整文件

- `lib/main.dart`
  - 梳理桌面端平台判定与主框架挂载
- `lib/pages/desktop_layout.dart`
  - 重做桌面框架容器、左导航栏和主区结构
- `lib/pages/chats.dart`
  - 去移动端 `AppBar` 化，改桌面列表面板
- `lib/pages/bots.dart`
  - 与 Chats 页统一桌面列表面板结构
- `lib/pages/chat.dart`
  - 重组聊天详情页容器层级
- `lib/pages/chat/message_list.dart`
  - 调整消息流宽度、间距、状态块样式
- `lib/pages/chat/message_input.dart`
  - 重构底部输入区桌面样式
- `lib/utils/theme.dart`
  - 可补充桌面样式 token、扩展 theme 或常量

### 12.2 实现方式建议

1. 将桌面端布局容器与移动端页面容器解耦。
2. 尽量保留现有状态管理和回调链路，仅重做外层承载结构。
3. 将颜色、圆角、padding 提炼为常量，避免分散在多个页面中。
4. 为列表项、状态卡片、桌面输入区抽共用组件。

## 13. 风险与注意事项

1. 当前页面大量使用 `Scaffold` 与 `AppBar`，桌面容器化改造时可能出现重复背景、重复安全区和双层滚动问题。
2. `SidebarX` 的扩展能力可能不足以还原目标样式，需预留自定义导航实现空间。
3. 聊天详情页如果仍沿用移动端消息布局，最终视觉会与 spec 偏差较大。
4. Windows/Linux 字体渲染存在差异，不能完全依赖固定高度，需要预留文本换行空间。

## 14. 验收标准

满足以下条件即可视为完成：

1. Windows/Linux 启动后，整体桌面端视觉明显接近参考图的工作台布局。
2. 左侧为窄导航栏，中间为列表栏，右侧为主内容画布。
3. 列表栏、详情区、输入区均采用统一圆角与浅色系卡片语言。
4. 聊天消息流存在最大宽度控制，阅读体验明显优于当前实现。
5. 状态信息可使用独立卡片样式展示，而非混杂在普通正文内。
6. 新建聊天、选择聊天、选择智能体、编辑智能体的既有流程仍可正常工作。
7. 移动端外观与交互无回归。

## 15. 建议拆分任务

建议按以下顺序实施：

1. 桌面设计 token 抽离
2. `DesktopLayout` 外框重构
3. Chats/Bots 列表面板样式统一
4. Chat 详情区容器重构
5. 消息流与状态卡片样式落地
6. 输入区样式重构
7. Windows/Linux 验收与微调

## 16. 产出物

本 spec 对应的最终产出应包括：

- Windows/Linux 桌面端统一样式实现
- 关键桌面组件的样式常量或主题抽取
- 至少一轮 Windows/Linux 实机或模拟环境视觉验收
